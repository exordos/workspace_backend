# Интеграция Workspace ↔ Zulip

Документ описывает, как должен работать мост между Workspace Messenger и Zulip.
Первая часть — поведение фонового сервиса `workspace-integration-bridge-worker`.

## Границы первой версии

Первая версия интеграции — полноценный двусторонний MVP, но не вся поверхность
Zulip API входит в обязательный scope.

В первой версии обязательно реализуются:

- inbound-синхронизация всей доступной истории Zulip;
- realtime inbound через Zulip event queue;
- public stream/topic;
- private 1:1;
- импорт пользователей из Zulip с сопоставлением с пользователями Workspace;
- импорт файлов и вложений в Workspace storage;
- outbound для сообщений: отправка, редактирование, удаление;
- outbound для реакций;
- outbound для read/unread через существующий механизм Workspace flags;
- диагностика через логи и таблицы БД;
- backend unit/integration tests.

В первой версии не входят:

- групповые private-чаты Zulip;
- outbound-управление stream/topic;
- outbound-подписки и отписки;
- outbound-изменение участников и ролей;
- отдельный UI, API или CLI статуса интеграции;
- обязательная приёмка через живой Zulip UI.

## workspace-integration-bridge-worker

### Назначение

`workspace-integration-bridge-worker` — долгоживущий фоновый процесс, который
синхронизирует данные между Zulip и Workspace Messenger в обе стороны:

- **Inbound (Zulip → Workspace):** вся доступная история, новые и изменённые
  потоки, топики, сообщения, реакции, файлы, read/unread-состояние и участники
  из Zulip попадают в модели Workspace и порождают события `m_workspace_events`
  для UI.
- **Outbound (Workspace → Zulip):** действия пользователя в Workspace с
  `source_name = zulip` доставляются обратно в Zulip от его имени: отправка,
  редактирование и удаление сообщений, реакции, read/unread. Управление
  стримами, топиками, подписками, участниками и ролями остаётся вне первой
  версии outbound.

Запуск:

```bash
workspace-integration-bridge-worker --config-file etc/workspace/workspace.conf
```

Сервис наследует `BasicService` и выполняет итерацию примерно раз в **3 секунды**

### Продуктовые правила первой версии

- Синхронизируется **вся доступная история Zulip**, а не только новые сообщения
  после подключения и не ограниченное окно.
- Если event queue Zulip протухла и старые события уже недоступны, мост
  восстанавливает **итоговое состояние** объектов, а не исторический event-log.
  Например, если реакцию успели поставить и снять до восстановления связи, в
  Workspace фиксируется только текущее отсутствие реакции.
- Для импортированных Zulip-сущностей источником правды считается Zulip.
  Workspace-действия отправляются обратно в Zulip, но при последующей
  пересинхронизации состояние Zulip имеет приоритет.
- Двусторонняя синхронизация первой версии включает сообщения, правки,
  удаления, реакции и read/unread. Переименования стримов и топиков, подписки,
  отписки, участники и роли синхронизируются inbound, но их outbound-изменения
  не являются частью первой версии.
- Групповые private-чаты Zulip в первой версии не синхронизируются.
  Поддерживаются публичные stream/topic и private 1:1.
- Если в истории встречается Zulip-пользователь, которого ещё нет в Workspace,
  Workspace-пользователь автоматически создаётся из Zulip-профиля.
- Файлы и вложения импортируются в Workspace storage. Content сообщений нужно
  процессить так, чтобы ссылки на Zulip uploads заменялись на нативные
  Workspace URN, например `urn:images:<uuid>` для изображений и аналогичные URN
  для других типов файлов.
- Состояние интеграции и ошибки первой версии видны через логи и таблицы в БД.
  Отдельный пользовательский или админский UI статуса не является требованием
  первой версии.
- Свежие сообщения должны появляться в Workspace до полной загрузки всей
  истории: агент выполняет задачи от новых `message_id` к старым, а старая
  история догружается фоном.
- Сообщения, импортированные из Zulip, назначаются реальным авторам в Workspace,
  как будто они были написаны в Workspace, но сохраняют признак источника
  `source_name = zulip` и source-связку с Zulip.
- Одно публичное Zulip-сообщение должно соответствовать одному Workspace-
  сообщению в рамках `(project_id, server_url, zulip_message_id)`, даже если
  его видят несколько подключённых аккаунтов.
- Удалённые Zulip-сообщения физически удаляются из Workspace. Если удаление
  инициировано в Workspace для Zulip-сообщения, outbound delete отправляется в
  Zulip от имени аккаунта пользователя, который выполнил удаление.

### Общая архитектура

В системе есть один bridge-сервис. Внутри него главный агент
(`WorkspaceIntegrationBridgeWorker`) управляет несколькими `ZulipBridgeWorker`.
Zulip-серверов может быть несколько; для каждого подключённого аккаунта агент
может поднять отдельного воркера подписки. Воркеры отвечают за чтение Zulip
event queue и передачу событий агенту, а синхронизация, порядок обработки задач
и запись состояния управляются агентом.

На каждой итерации главный агент выполняет `_start_bridges()`:

1. Загружает из базы все `ExternalAccount` с `account_type = zulip`, у которых
   заполнены `account_settings.credentials`. Аккаунты без credentials пропускаются.
2. Для каждого такого аккаунта вычисляет ключ воркера
   `(project_id, server_url, user_uuid)` и проверяет, есть ли уже запущенный
   `ZulipBridgeWorker` с этим ключом. Если нет — создаёт поток, передаёт ему
   общую очередь `_sync_queue` (воркер → агент), вызывает `worker.start()`.
3. Воркер подписывается на event queue Zulip: читает сохранённый `queue_id` из
   `ZulipEventQueueState` и пытается продолжить работу с той же очередью. Если
   очередь доступна, воркер продолжает читать события с сохранённого
   `last_event_id` без полного resync. Если подписки ещё не было
   (`queue_id` пустой) или старая очередь недоступна, воркер регистрирует новую
   через Zulip API. После этого кладёт в `_sync_queue` событие
   `ZulipEventQueueCreated` — сигнал, что подписка на новую очередь только что
   создана (вместе с `UpdateZulipQueueState` для сохранения `queue_id` и
   `last_event_id` в базе).

У агента должна быть возможность остановить любой `ZulipBridgeWorker` в любой
момент через `worker.stop()`. Воркер выставляет флаг `_stopped` и выходит из
циклов синхронизации. После вызова `worker.stop()` воркер должен завершить
работу не дольше чем за **10 секунд**.

Отсюда следует, что ожидание ответа от Zulip на long-poll (`get_events`) тоже
не должно превышать **10 секунд** — иначе воркер не сможет вовремя проверить
флаг `_stopped` и завершиться.

### Задачи синхронизации истории

Если воркер смог продолжить работу с той же event queue, агент не пересоздаёт
history-задачи и продолжает обычную обработку.

Когда агент получает от воркера `ZulipEventQueueCreated`, подписка на новую
event queue только что создана — первое подключение или reconnect, при котором
старая очередь уже недоступна. За время отсутствия связи могло произойти что
угодно: новые сообщения, правки текста, удаления, реакции, read/unread.

Агент **не просматривает** существующие задачи и не ищет пробелы. Он:

1. **удаляет** все несинхронизированные задачи (`pending`, `failed`) для этого
   external account;
2. узнаёт актуальный последний `message_id` на Zulip-сервере;
3. **создаёт заново** задачи на синхронизацию **всех** сообщений (по 100 на
   задачу).

Полагаться на частично выполненные задачи нельзя — после дисконнекта состояние
на сервере Zulip уже могло измениться.

Такой resync не пытается восстановить старые события из протухшей event queue.
Zulip API даёт текущее состояние сообщений, поэтому агент должен привести
Workspace к текущему состоянию Zulip: текст, удаление, флаги, реакции, файлы,
read/unread, имена стримов/топиков и bindings. Исторические промежуточные
изменения, которые уже исчезли из event queue, не восстанавливаются.

**Приоритет по `message_id`.** Сообщения с более высоким id всегда
приоритетнее сообщений с более низким id. Это правило действует везде: при выборе
задачи синхронизации истории, при обработке событий из очереди воркера, при
конкуренции realtime-событий и history sync за один цикл агента.

#### Что такое задача

Задача хранится в таблице `m_zulip_history_sync_tasks`, модель —
`ZulipHistorySyncTask` (`workspace/messenger_api/dm/models.py`), миграция —
`migrations/0087-add-zulip-history-sync-tasks-ad1c90.py`.

Задача описывает интервал сообщений, который нужно загрузить из Zulip:

- **с какого сообщения** — `from_message_id`;
- **по какое** — `to_message_id`;
- **для какого пользователя** — `user_uuid`;
- **какой внешний аккаунт** — `external_account_uuid`, `server_url`, `project_id`;
- **статус выполнения** — `pending`, `done`, `failed`:
  - `pending` — задача создана, ещё не выполнялась;
  - `done` — интервал `from_message_id`…`to_message_id` полностью синхронизирован;
  - `failed` — выполнение завершилось ошибкой, подробности в `last_error`;
- **даты** — `created_at`, `updated_at`;
- **ошибка** — `last_error` (если была).

Ограничение в **100 сообщений** на интервал действует только для задач в статусе
`pending`. Соблюдать его при создании задачи должен тот, кто её создаёт (агент
при первичной/повторной синхронизации, воркер при фиксации realtime-сообщения).

#### Создание задач

Пример: на Zulip-сервере сейчас 600 сообщений.

Агент создаёт **6 задач** по 100 сообщений:

| Задача | from_message_id | to_message_id |
|---|---|---|
| 1 | 0 | 99 |
| 2 | 100 | 199 |
| 3 | 200 | 299 |
| 4 | 300 | 399 |
| 5 | 400 | 499 |
| 6 | 500 | 599 |

Для каждого `external_account` — свой набор задач. Тот же принцип применяется
и при первом подключении, и после каждого `ZulipEventQueueCreated`.

При выборе задачи на выполнение агент берёт `pending`-задачу с **наибольшими**
`from_message_id` / `to_message_id` — сначала свежие сообщения, затем более
старые интервалы.

#### Задачи от realtime-событий

Когда воркер получает новое сообщение через event queue и обработал его, он
тоже создаёт задачу — фиксацию уже обработанного интервала. Например, для
пользователя 1: `from_message_id = 55`, `to_message_id = 55`.

Такие задачи будут мелкими (часто по одному message id). Отдельная часть кода
позже будет **объединять** их в более крупные задачи.

#### Один цикл агента

За одну итерацию агент:

1. обрабатывает **одну** задачу синхронизации истории из базы — с наибольшим
   `message_id` среди `pending`;
2. обрабатывает **все** события, накопившиеся в `_sync_queue` от воркеров
   (внутри очереди — тоже от более новых id к более старым).

#### Повторная обработка сообщений

Во время синхронизации истории или разбора очереди сообщение могло быть
обработано раньше — например, через realtime до reconnect или при пересечении
задач. Такое сообщение **нельзя просто пропускать**: его нужно снова прогнать
через логику создания/обновления, чтобы актуализировать состояние в Workspace
(текст, read-флаг, реакции и т.д.).

Маппинг «Zulip `message_id` → сообщение в Workspace» хранится отдельно, чтобы
находить уже существующую запись и идти по пути **update**, а не create. Для
публичных сообщений уникальный ключ маппинга:
`(project_id, server_url, zulip_message_id)`.

**Без лишних уведомлений.** Хелперы обновления должны быть идемпотентными:
их могут вызвать повторно, когда само сообщение для пользователя не менялось.
Если новых данных нет (тот же content, те же флаги, та же реакция), обновление
завершается **без** событий в `m_workspace_events` — пользователю не должно
приходить уведомление об изменении. Это техническая пересинхронизация, а не
реальное редактирование в Zulip.

Если данные всё же отличаются — применить изменение и породить обычное событие
для UI.

При конфликте данных Zulip и Workspace для imported-сущностей побеждает Zulip.
Локальное Workspace-действие должно быть доставлено в Zulip через outbound-путь;
если Zulip его принял, оно подтвердится последующей синхронизацией. Если текущее
состояние Zulip отличается от локального, Workspace приводится к Zulip.

#### Что обрабатывается по каждому сообщению

Для **каждого** inbound-сообщения обязательно прогонять полный набор данных
из Zulip, а не только текст:

- **content** — тело сообщения;
- **файлы** — вложения скачиваются из Zulip, сохраняются в Workspace storage, а
  ссылки в content заменяются на Workspace URN (`urn:images:<uuid>` и аналоги);
- **флаги** — в том числе `read` и другие flags из event/message payload; для
  read/unread используется существующий механизм Workspace flags, которые
  создаются отдельно для каждого пользователя;
- **реакции** — добавление и снятие;
- **удаление** — если сообщение помечено удалённым в Zulip, соответствующее
  Workspace-сообщение удаляется физически.

Пропускать отдельные части нельзя: при повторной обработке нужно сверить всё
с текущим состоянием в Workspace.

Если вложение из Zulip не удалось скачать или сохранить, сообщение всё равно
создаётся или обновляется. Вместо ссылки на файл в content вставляется ссылка на
заранее подготовленный шаблон-заглушку, который сообщает, что файл скачать не
удалось. Ошибка сохраняется в диагностическом `last_error`/status-поле, чтобы
её можно было увидеть через БД и логи.

#### Переименование стрима и топика

При обработке сообщения нужно смотреть не только на `stream_id` и `topic_name`
в source, но и на **актуальные имена** из Zulip:

- если **имя стрима** (`display_recipient` / name в Zulip) изменилось — обновить
  стрим в Workspace;
- если **имя топика** (`subject` в Zulip) изменилось — обновить топик в
  Workspace.

Поиск стрима и топика по-прежнему идёт по source, а не по отображаемому имени.
Смена имени — это реальное изменение: его нужно записать и, если данные
действительно изменились, породить событие для UI. Если имя совпадает с уже
сохранённым — без лишних уведомлений, по тем же правилам идемпотентности.

Для private-сообщений имя топика в source остаётся `"zulip"`; переименование
пользователем в Workspace не отменяет это правило поиска по source.

### Синхронизация стримов и топиков

Сообщение из Zulip нельзя записать в Workspace, пока не существуют соответствующие
стрим и топик. При обработке inbound-сообщения в рамках задачи воркер должен
пройти цепочку: найти или создать стрим → найти или создать топик → создать
сообщение.

#### Если стрим ещё не существует

Когда сообщение ссылается на стрим, которого ещё нет в Workspace, его нужно
создать до записи сообщения.

**Публичный стрим.** Из одного сообщения недостаточно данных: нет описания,
создателя и полного списка участников. Воркер, выполняя задачу синхронизации,
перед продолжением обработки этого сообщения должен запросить у Zulip метаданные
стрима и список подписчиков, создать стрим в Workspace (и топик, если его ещё
нет), и только после этого продолжить выполнение задачи.

**Личный стрим (private).** Данных в самом сообщении достаточно: id чата,
участники, тип переписки. Стрим создаётся сразу, без отдельного запроса
списка всех стримов сервера. Топик для таких сообщений создаётся с именем
`"zulip"`.

По числу участников:

- **два** — личная переписка один на один;
- **больше двух** — пропускаем (групповые private-чаты не синхронизируются в
  первой версии);
- **меньше двух** — сообщение пропускается.

#### Создание стрима в Workspace

Создатель стрима в Zulip, автор сообщения и любой другой участник, без которого
нельзя создать стрим, сообщение или bindings, должен быть сопоставлен с
пользователем Workspace.

Порядок сопоставления:

1. найти пользователя через привязанный Zulip-аккаунт по
   `(server_url, zulip_user_id)`;
2. если привязки нет — найти Workspace/IAM-пользователя по email;
3. если пользователь не найден — автоматически создать импортированного
   Workspace-пользователя из Zulip-профиля.

Zulip `user_id`, email, имя, avatar/source-данные сохраняются так, чтобы
последующие сообщения этого пользователя связывались с той же Workspace-записью.

IAM остаётся внешним источником корпоративных пользователей, но отсутствие
записи в IAM не блокирует импорт истории Zulip. Если позже появится каноническая
IAM-запись для того же пользователя, отдельная логика сопоставления должна
связать её с уже созданным Workspace-пользователем, а не создавать дубль.

В Workspace сохраняются имя, описание, флаги доступа, привязка к Zulip
(`stream_id`, `server_url`, `source_name = zulip`).

**Роли.** Создатель стрима в Zulip становится **owner** стрима в Workspace.
Все остальные известные подписчики из Zulip получают роль **member**. Owner
не дублируется в списке members.

#### Топики

Отдельной предварительной синхронизации списка топиков из Zulip не требуется.
Топик появляется в Workspace, когда приходит первое сообщение, которое на него
ссылается.

Искать и сопоставлять топик нужно по **source** (`stream_id`, `server_url`,
`topic_name` из Zulip), а не по отображаемому имени в Workspace — пользователь
может переименовать топик локально, и имя перестанет совпадать с `subject` из
Zulip.

**Личные сообщения.** Для одного external account в private-стриме может быть
только **один** топик — в source всегда `topic_name = "zulip"`. Отображаемое
имя пользователь может переименовать; для поиска топика используется только
source, а не текущее имя в Workspace.

**Публичные стримы.** Ограничения «один топик на аккаунт» нет: в одном стриме
может быть сколько угодно топиков, каждый определяется своим `subject` из Zulip.

#### Когда создаётся или обновляется стрим

Стримы могут появляться по мере обработки сообщений или через отдельные
stream/subscription-события Zulip. При обработке истории сообщение всё равно
должно уметь создать недостающий стрим и топик:

| Ситуация | Что делать |
|---|---|
| В задаче встретилось сообщение в неизвестный публичный стрим | Воркер запрашивает стрим и подписчиков у Zulip, создаёт стрим и топик, продолжает задачу |
| В задаче встретилось сообщение в личный чат | Воркер собирает стрим из данных сообщения, создаёт топик `"zulip"`, продолжает задачу |

После создания или обновления стрима нужно актуализировать bindings участников:
новые подписчики из Zulip добавляются как members, отписанные участники удаляются
из bindings или помечаются неактивными согласно модели Workspace.

### Outbound-синхронизация

Для объектов с `source_name = zulip` Workspace должен отправлять пользовательские
действия обратно в Zulip.

В первой версии outbound включает:

- отправку новых сообщений в stream/topic и private 1:1;
- редактирование и удаление сообщений;
- добавление и снятие реакций;
- read/unread-действия через существующие Workspace flags.

В первой версии outbound не включает:

- переименование stream/topic;
- подписки и отписки;
- изменение участников и ролей там, где это поддерживается Zulip API.

Outbound-действие выполняется от имени Workspace-пользователя, который совершил
действие, через его подключённый Zulip-аккаунт. Например, удаление сообщения в
Workspace отправляется в Zulip от аккаунта пользователя, который удалил
сообщение.

Outbound-действие считается окончательно применённым только после успешного
ответа Zulip или после того, как inbound/resync подтвердил соответствующее
состояние.

Если Zulip отклонил действие из-за прав доступа или изменившегося состояния,
действие переводится в `failed`. Состояние Workspace затем приводится к Zulip
через inbound/resync. Заведомо невозможные действия не должны бесконечно
повторяться.

### Диагностика

В первой версии отдельный UI статуса интеграции не требуется. Операционное
состояние должно быть диагностируемо через:

- логи `workspace-integration-bridge-worker`;
- `ZulipEventQueueState`;
- `ZulipHistorySyncTask`;
- outbound retry/failed-состояния;
- ошибки импорта файлов и пользователей, сохранённые в соответствующих
  `last_error`/status-полях.

### Критерии приёмки первой версии

Первая версия считается готовой, если покрыта backend unit/integration tests.
Обязательные группы проверок:

- первичное создание задач истории и выбор задач от свежих сообщений к старым;
- продолжение работы при reconnect на ту же Zulip event queue;
- создание новой queue и пересоздание незавершённых history-задач;
- идемпотентный импорт одного публичного сообщения несколькими аккаунтами без
  дублей;
- сопоставление Zulip-пользователя по привязке, затем по email, затем создание
  импортированного пользователя;
- импорт public stream/topic;
- импорт private 1:1 и пропуск групповых private-чатов;
- импорт файлов в Workspace storage и замена ссылок на Workspace URN;
- вставка ссылки на шаблон-заглушку при ошибке скачивания файла;
- read/unread через Workspace flags по пользователям;
- физическое удаление Workspace-сообщения при удалении в Zulip;
- outbound send/edit/delete;
- outbound add/remove reaction;
- outbound read/unread;
- перевод outbound-действия в `failed`, если Zulip отклонил его из-за прав или
  состояния.
